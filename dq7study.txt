Reverse Engineering the Monolithic Engine of Dragon Warrior VII: Memory Partitioning, Event Flag Matrices, and Asset Compression ProtocolsMemory Partitioning and Heap Allocation MappingThe execution of a massive, multi-disc role-playing game on the fifth-generation PlayStation platform forced developer Heart Beat to bypass standard operating system APIs in favor of a custom, highly optimized system architecture. With only $2\text{ MB}$ ($2048\text{ KB}$) of system RAM available, the main executable SLUS_012.06 (and its Japanese counterpart SLPM_865.00) maps out a strict partitioning layout. This layout separates the static, resident "base engine" from volatile "expansion content" and high-bitrate data streamed dynamically from the optical disc.The lower bounds of system memory host the resident base engine, which includes core MIPS R3000 CPU instructions, geometry pipeline interfaces, peripheral input maps, and the main gameplay execution loop. Immediately above these system bounds lies the game state tracking matrix, followed by dedicated sound processing software. Because the standard Sony PsyQ sound library was too generic and resource-intensive, Heart Beat implemented a custom sound driver, designated SEQq, which utilizes the signature "qQES". This custom driver bypasses the standard SEQ/VAB protocols to optimize the utilization of both system RAM and Sound RAM (SPU WRAM).PlayStation System Memory Map LayoutThe partition boundaries mapped within the PlayStation's $2\text{ MB}$ physical memory space are structured as follows:Physical Address RangeSegment ClassificationCore Engine Function / SubsystemCitations0x80000000 - The central function that coordinates this operation is located at address 0x80076040 (open_file). This routine acts as an interface to the PlayStation's CD-ROM controller. When the player transitions between map sectors, open_file invokes a secondary, non-blocking sub-loader at address 0x80082598 (mopen_file).The mopen_file routine operates as follows:It accepts a single parameter passed through CPU register $a0, which points to a file descriptor struct. This struct contains the target Logical Block Number (LBN), sector length, and the destination address in the dynamic heap.Rather than attempting to copy large data blobs in a single processing frame—which would stall the main game loop—the function reads the file from the disc incrementally, sector-by-sector, in $2048\text{-byte}$ chunks.For map instances, the sub-loader detects the resource type as 0x13, prompting the engine to partition the dynamic loading process across multiple frames.The system leverages DMA Channel 4 (the CD-ROM direct memory access pathway) to stream sectors directly into the dynamic map geometry heap starting at 0x80138000, bypassing main CPU cycles.This streaming approach has a noticeable impact on emulators. When the game menu is opened, the engine takes a snapshot of the current frame buffer from VRAM and copies it to a dedicated background buffer in System RAM. Because of strict memory constraints, this copy is captured at the game's native internal rendering resolution (typically $256\times224$ or $320\times240$ pixels). On original hardware, this transfer is seamless. However, on modern emulators running at upscaled resolutions, this process creates a noticeable rendering discrepancy: the high-resolution menu elements are overlaid on top of a low-resolution, blocky background snapshot, resulting in a distinct "blurry" menu effect.Event Trigger and Flag Matrix ArchitectureTo maintain a persistent, non-linear narrative across dozens of islands existing in distinct past and present time periods, the Heart Beat engine utilizes a compact, highly dense game state tracking system. This tracking is managed by an event flag matrix residing in the system's static memory bounds.Tracing the Progression Flags in RAMLive memory tracing on a PlayStation emulator locates the core game progress flags within the RAM range 0x8000F800 through 0x80010000. This space acts as a bit-packed matrix that records every major progression state :Active system flags are modified by the engine's state-writing routines. For example, the "Save Anywhere" debug codes write directly to the flag ranges 0x8000F83C through 0x8000F846.Equipment rules and character restrictions are managed by related addresses in this block; for instance, the "Equip Any Armor" bypass writes directly to static address 0x80069BCA to override standard inventory validation checks.To trace these state changes in a debugging environment (such as No$psx), developers can set a write breakpoint directly on the target flag address using the syntax [0x8000F83C]!. This file spans several hundred megabytes and is divided into strict $2048\text{-byte}$ blocks to align perfectly with the physical sectors of the PlayStation's CD-ROM drive.Archive Structural LayoutThe HBD1PS1D file format uses a structured hierarchy designed for high-performance optical seeking, divided into distinct block classifications :Header BlockThe first $2048\text{-byte}$ sector functions as the primary volume descriptor. The ASCII string hdb1ps1d.q71 is located at byte offset 0x400 ($1024\text{ bytes}$), serving as the game's archive identification signature.Primary Data Blocks (* 00 00 00 Blocks)These blocks act as index directories for grouped assets. Each directory block begins with a $16\text{-byte}$ header :Offset 0x00 (4 bytes): Sub-block Count. The unpadded byte length of the raw, concatenated sub-block payload.Offset 0x0C (4 bytes): Reserved. Structured as a null dword (0x00000000).Asset Sub-blocksEach primary data block contains contiguous asset payloads. The decompressor initializes a sliding dictionary ring buffer of $4096\text{ bytes}$. This buffer is initialized entirely with zero (0x00) bytes.The input stream is read as a sequence of control blocks, starting with a $1\text{-byte}$ control word, $C$. The decompressor parses this control byte bit-by-bit from the least significant bit ($b_0$) to the most significant bit ($b_7$) to process the compressed payload :$$\text{Bit } b_i \text{ of control byte } C \implies \begin{cases} 1 & \text{Literal Byte Copy} \\ 0 & \text{Dictionary Reference (16-bit Descriptor } W) \end{cases}$$Literal Byte Copy ($b_i = 1$)The decompressor reads the next raw byte directly from the input stream, writes it to the destination output buffer in RAM, and appends it to the sliding dictionary history window:$$\text{Output} \leftarrow \text{Input}[I]$$$$D \leftarrow D + 1, \quad I \leftarrow I + 1$$Dictionary Reference ($b_i = 0$)The decompressor reads a $2\text{-byte}$ ($16\text{-bit}$) descriptor word, $W$, from the input stream:$$W \leftarrow (\text{Input}[I] \ll 8) \mid \text{Input}[I+1]$$$$I \leftarrow I + 2$$This $16\text{-bit}$ word is unpacked to extract the history ring-buffer offset, $O$, and the repeating match length, $L$:$$O \leftarrow W \gg 4$$$$L \leftarrow (W \ \& \ \text{0x0F}) + 3$$The decompressor then copies $L$ bytes from the sliding history buffer starting at offset $O$ directly into the active destination RAM buffer, updating the dictionary ring buffer with the copied bytes. Heart Beat's decision to use the LZSS0 algorithm represents a deliberate technical trade-off. Because the PlayStation's CPU lacks a dedicated hardware decompression coprocessor (unlike modern systems or its video-only MDEC chip), on-the-fly decompression of highly packed assets introduces significant processing overhead. LZSS0 requires minimal CPU cycles for decompression, relying on simple bitwise shifts and memory copies. By accepting slightly lower compression ratios compared to more advanced algorithms, the engine is able to decompress graphics, scripts, and map data on-the-fly without causing frame-rate stutter or disrupting the interrupt-driven sound processing threads.Isometric Dungeon Tile-set Decompression and AssemblyThis decompression pipeline is essential for assembling the game's isometric dungeon environments. These environments are built using a collection of $16\times16\text{ pixel}$ tiles, with walls often rendered at a height of $32\text{ pixels}$ to establish isometric depth and perspective.The dungeon assembly sequence proceeds as follows :The engine calls open_file to locate the target dungeon directory block inside HBD1PS1D.Q71.Two distinct sub-blocks are loaded: one containing the compressed $16\times16$ texture tile-sets, and another containing the grid-based isometric layout map.The texture payload is processed by the LZSS0 decompressor, which inflates the compressed data into a raw, linear $4\text{-bit}$ or $8\text{-bit}$ indexed bitmap array representing color indices mapped to local palette tables.The layout sub-block is decompressed to reconstruct a two-dimensional grid of cell pointers. Each cell pointer is a $16\text{-bit}$ descriptor containing the following structural metadata:Bits 0-9: Tile Index (points directly to the decompressed $16\times16$ texture array).Bit 10: Horizontal Texture Flip Flag (toggles horizontal mirroring).Bit 11: Vertical Texture Flip Flag (toggles vertical mirroring).Bits 12-15: Elevation and Height Parameters (defines the isometric layer coordinate, determining whether the tile is rendered at base level or elevated to form wall structures).The rendering engine parses this layout grid sequentially. To construct the isometric perspective, any tile containing an active elevation parameter has its vertical draw coordinate offset by a fixed rate of $4$ to $6\text{ pixels}$. This adjustment ensures that characters and objects align correctly with the floor tiles rather than floating on edge boundaries, creating an accurate sense of scale and depth within the engine's isometric environments.Technical Synthesis of Core SystemsThe custom engine developed by Heart Beat for Dragon Warrior VII shows how software-level optimizations can overcome the physical limitations of legacy hardware. This structural architecture functions as an integrated pipeline :File operations are organized into sector-aligned physical blocks to maximize CD-ROM drive efficiency and reduce head seek times.High-bitrate data streaming is controlled by the asynchronous mopen_file routine. This allows the engine to load dynamic map blocks in the background while keeping the main frame loop and interrupt-driven audio playback threads running smoothly.Progression tracking is consolidated into a compact event flag matrix, minimizing memory usage. These flags are evaluated in real-time during map transitions, allowing the engine to selectively stream only the required 3D models and character assets into the active heap.CPU cycle stalls are minimized by using the computationally simple LZSS0 compression algorithm, allowing real-time asset decompression to run alongside the game's core gameplay processes.By combining these memory-saving techniques, Heart Beat engineered a highly efficient platform for its scale. This dynamic allocation system allowed the studio to bypass the hardware limitations of the fifth generation of consoles, fitting an exceptionally large and complex RPG world onto the PlayStation.
